# Piano Learning PWA — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** PWA d'apprentissage du piano hébergée sur GitHub Pages : notes qui tombent + partitions, feedback micro/MIDI/toucher, bibliothèque + import, progression locale.

**Architecture:** App statique Vite+TypeScript sans framework. Un modèle `Song` unique alimente deux vues (canvas falling-notes, OSMD sheet). Un `scheduler` pilote la lecture ; un `matcher` compare notes attendues vs jouées (sources : mic FFT, Web MIDI, toucher). Persistance localStorage/IndexedDB. Aucune dépendance réseau après chargement (PWA offline).

**Tech Stack:** Vite, TypeScript, Vitest, OpenSheetMusicDisplay, @tonejs/midi, vite-plugin-pwa, Canvas 2D, Web Audio API, Web MIDI API.

## Global Constraints

- Langue de l'UI : **français**.
- Cible : Chrome Android (portrait ET paysage) + Chrome desktop. Tactile d'abord.
- 60 fps sur canvas mobile → pas d'allocation dans la boucle rAF, redraw partiel du clavier.
- Tout asset (samples, morceaux, fonts) est bundlé dans le repo — zéro CDN à l'exécution.
- `base` Vite = `./` (chemins relatifs pour GitHub Pages sous /<repo>/).
- Tests : Vitest, la logique `src/core/**` et `src/audio/mic-*` doit être couverte.
- Commits fréquents, messages en français, préfixes feat/fix/chore/test.

### Task 5: Scheduler (core/scheduler.ts)

**Files:**
- Create: `src/core/scheduler.ts`
- Test: `tests/scheduler.test.ts`

**Interfaces:**
- Produces:

```ts
export interface SchedulerOpts {
  song: Song; speed: number;           // 0.25..1.5
  waitMode: boolean;                    // pause sur notes attendues
  loop?: [number, number] | null;       // secondes (temps référence)
  hands: 'left' | 'right' | 'both';
  onNotesDue(notes: SongNote[]): void;  // notes atteignant la tête de lecture
  onEnd(): void;
}
export class PlaybackScheduler {
  constructor(opts: SchedulerOpts);
  start(from?: number): void; pause(): void; stop(): void;
  seek(t: number): void;
  get time(): number;                   // temps courant (réf.), calculé via injection d'horloge
  get waiting(): boolean;               // true si waitMode bloque
  satisfy(midi: number): void;          // note jouée correcte -> peut débloquer
  tick(nowMs: number): void;            // avancée pilotée de l'extérieur (rAF/test)
}
```

Horloge injectée via `tick(nowMs)` → 100 % testable sans timers. En waitMode : la tête s'arrête sur chaque groupe de notes (même time à ±50 ms, filtré par `hands`) jusqu'à ce que tous les `midi` du groupe soient `satisfy()`. Boucle : à `loop[1]`, seek à `loop[0]`. Tests : progression au speed, groupe d'accord en attente, satisfy partiel/complet, boucle, fin. Commit.
